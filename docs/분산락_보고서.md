## 1. 분산환경 문제

다수의 서버와 다수의 DB가 존재하는 분산 환경에서 트랜잭션만으로는 한계가 존재합니다.
동시성 문제가 발생했을 때 데이터 무결성 문제가 발생할 수 있습니다.
단일 DB를 사용하더라도 분산락을 사용해야 하는 이유는 다음과 같습니다.

* **분산락을 먼저 사용해 DB 경합 충돌, 데드락 방지로 DB의 부하 감소**

현재 문서에서는 다수의 서버와 단일 DB를 사용한다고 가정하고 분산락을 사용해야 하는 이유를 설명합니다.

## 2. 분산락 처리 방법

분산락의 구현은 크게 3가지로 나눌 수 있습니다.

1. Simple Lock
2. Spin Lock
3. Pub/Sub Lock
   각 방법에 대해서 설명합니다.

| 구현 방법        | 설명                                                       | 장점                                | 단점                                                                                 |
|--------------|----------------------------------------------------------|-----------------------------------|------------------------------------------------------------------------------------|
| Simple Lock  | Redis의 SET NX 명령어를 사용하여 분산락을 구현합니다.                      | 구현이 간단하고 부하가 상대적으로 적습니다.          | 락 획득 실패 시 재시도를 하지 않습니다. 경합이 치열한 경우 다수가 실패할 수 있습니다.                                 |
| Spin Lock    | Redis의 SET NX 명령어를 반복 사용하여 분산락을 구현합니다.                   | 락을 획득이 실패하더라도 일정 횟수는 재시도할 수 있습니다. | 락 획득 후 로직의 수행 시간이 길다면 락 획득 시도가 많아집니다.                                              |
| Pub/Sub Lock | Redisson 라이브러리에서 지원하는 Redis Pub/Sub 기능을 사용하여 분산락을 구현합니다. | 락 해제 시 알림을 받아 불필요한 재시도를 줄일 수 있습니다. | 트래픽이 몰리게 된다면 구독자가 많아져 다수의 커넥션을 지속해야할 수 있습니다. 락 해제 메시지가 유실되어 계속 구독 상태에 머무를 수도 있습니다. |
